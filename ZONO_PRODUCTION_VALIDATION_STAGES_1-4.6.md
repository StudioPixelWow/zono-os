## 1. THE HEADLINE FINDING вАФ the kernel works; flow *coverage* does not

The event pipeline (emit вЖТ outbox вЖТ subscribers вЖТ timeline вЖТ graph вЖТ search вЖТ memory вЖТ recommendation вЖТ daily вЖТ executive вЖТ ask-context вЖТ ask-persistence) is **architecturally complete, idempotent, and failure-isolated for every event that is actually emitted.** That machinery is production-grade.

**But only 10 of the 24 required business flows emit a domain event today.** The other 14 have event types *defined in the registry* (`src/lib/kernel/events.ts`) but the `emitBusinessEvent(...)` call is **not wired at the write site**. For those flows, nothing flows down the pipeline from the write вАФ timeline, graph, search, and memory never see them.

This is the single most important production gap. The architecture is real; its **coverage is ~42%**.

## 4. Failure isolation (code-verified) вАФ вЬУ

- Processor: per-row `try/catch` (dead-letter after 5 retries) + per-subscriber `try/catch` (timeline primary lands first; 6 secondaries best-effort).
- Assembler `settle`: every layer independently caught вЖТ `failedLayers`; `truthLine` derived locally so **truth survives even if all other layers fail.**
- Surface grounding (Broker Brain/Daily/Executive/grounded-explanation): all `.catch(() => null)`.
- Ask persistence: fire-and-forget + internally never-throwing.
- **No fabrication:** honest empty/`null`/`"insufficient"` on failure. Two benign display-defaults flagged (rec confidence defaults 60/65) вАФ recommend surfacing as "unknown".

## 5. Security (code-verified) вАФ вЪ† ONE HIGH-SEVERITY BLOCKER

- Cross-org isolation вЬУ (every RLS policy filters `= current_org_id()`; every service-role write threads `org_id`).
- Public Ask / public sites / public search вЬУ (public_site mode + `FORBIDDEN`-field redaction; portals client-scoped; none read memory/notes/deals).
- Executive mode excludes broker-private memory at the assembler layer вЬУ.

**рЯФі BLOCKER вАФ broker-private memory isolation is defeated at the RLS layer.** `20260907120000_qa1_rls_coverage.sql` adds `ai_memory_qa1_read` (SELECT, `organization_id = current_org_id()` only). Postgres OR-combines permissive policies, so it **overrides** the private gate in `ai_memory_select`. Net: **any org member can read any other broker's `visibility='private'` / `scope_type='user'` memory in the same org.** Read paths (`ai-memory/repository.ts::listMemories`, `memory-canonical/read.ts::getUserMemory`) rely on RLS and add no `user_id` filter вЖТ reachable in app code. Contradicts the Batch 4.5 broker-private guarantee.
_Fix: drop or re-scope `ai_memory_qa1_read`; add `user_id`/visibility filters in the two read paths as defence-in-depth._
